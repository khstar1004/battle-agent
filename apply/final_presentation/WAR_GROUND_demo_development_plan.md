# WAR GROUND 데모 개발 상세 계획

## 1. 개발 목표

발표자료의 데모 흐름을 실제로 시연 가능한 웹 데모로 구현한다.

목표는 완전한 군사용 운영체계가 아니라, 심사 발표에서 **3분 30초 이내에 안정적으로 보여줄 수 있는 세미 라이브 데모 MVP**를 만드는 것이다.

데모의 핵심 메시지는 아래 한 문장이다.

> 계획서를 넣으면, 가상부대가 먼저 수행하고, 실패경로와 결심카드가 나온다.

## 2. 데모 시연 시나리오

발표에서 사용할 시나리오는 고정한다.

- 시나리오: 대대급 야간 기동훈련
- 임무: 제한 시간 내 지정 집결지 도착
- A안: 최단 경로 신속 전개
- B안: 우회 경로 안정 전개
- C안: 단계별 전개 후 예비대 대기
- 제한사항:
  - 통신 음영
  - 보급대기점 부족
  - 차량 정비 제한
  - 새벽 안개
  - 후송로 제한
- 최종 권고:
  - B안 우선
  - A안은 조건부 보완

## 3. 최종 데모 사용자 흐름

발표자는 아래 순서로 시연한다.

1. 앱 메인 화면 진입
2. `목업 작전계획 로드` 버튼 클릭
3. 작전계획, 방책, 제한사항이 입력되는 화면 확인
4. `가상부대 생성` 버튼 클릭
5. 지휘관·참모·현장수행·환경·적군 에이전트 생성 확인
6. `수행 리허설 실행` 버튼 클릭
7. 시간순 이벤트 재생
8. 실패경로 TOP 4 확인
9. 지식그래프/근거 연결 확인
10. 결심카드와 단편명령 초안 확인

데모 시간은 3분 30초 이내로 제한한다.

## 4. 개발 원칙

### 4.1 발표용 MVP 원칙

1차 데모는 실제 AI 추론을 매번 호출하지 않는다.  
발표장에서 네트워크, API, 응답 지연, 모델 환각 문제가 생기면 위험하다.

따라서 1차 MVP는 아래 방식으로 구현한다.

- 입력 데이터는 목업 파일로 고정
- 에이전트 생성 결과는 사전 정의된 JSON 사용
- 수행 리허설 이벤트는 결정론적 타임라인으로 재생
- 실패경로와 결심카드는 규칙 기반으로 생성
- LLM 호출은 선택 기능으로만 분리
- 인터넷 없이 로컬에서 실행 가능해야 함

### 4.2 심사 설득 원칙

데모는 기능 자랑이 아니라 스토리로 보여준다.

- “파일 업로드가 됩니다”보다 “계획서가 가상부대로 바뀝니다”를 보여준다.
- “AI가 답합니다”보다 “시간순으로 마찰이 드러납니다”를 보여준다.
- “위험요소가 있습니다”보다 “실패가 이어지는 경로가 보입니다”를 보여준다.
- “명령을 자동 생성합니다”가 아니라 “지휘관 검토용 결심카드를 제공합니다”로 보여준다.

## 5. 추천 기술 구조

현재 로컬 프로젝트는 정적 웹앱 구조다.

- `index.html`
- `app.js`
- `styles.css`
- `mock-data/`
- `assets/`

1차 MVP는 이 구조를 유지하는 것이 가장 안전하다.

### 5.1 1차 데모 구조

```text
index.html
app.js
styles.css

mock-data/
  operation_plan.json
  coa_options.json
  agents.json
  event_timeline.json
  failure_paths.json
  evidence_map.json
  decision_card.json
  fragmentary_order.json

assets/
  tactical-command-bg.png
  ontology-graph-bg.png
  failure-map-bg.png
```

### 5.2 2차 확장 구조

실제 기술 검증 단계에서는 아래 구조로 확장한다.

```text
frontend/
  React or plain JS UI

backend/
  FastAPI or Node API
  scenario_loader
  document_extractor
  agent_factory
  rehearsal_engine
  risk_fusion_engine
  evidence_linker
  decision_card_generator

data/
  deidentified_plans
  sop_snippets
  aar_samples
```

하지만 발표용 1차 데모에서는 백엔드를 새로 만들 필요가 없다.

## 6. 핵심 모듈 설계

## 6.1 Scenario Loader

목표: 목업 작전계획 데이터를 한 번에 로드한다.

입력:

- 작전명
- 임무
- 방책 A/B/C
- 제한사항
- 병력·장비
- 보급·정비
- 기상
- 통신 커버리지
- SOP 발췌

출력:

- 화면의 입력 테이블
- 작전 그래프 seed
- 에이전트 생성 seed

개발 항목:

- `mock-data/operation_plan.json` 생성
- `mock-data/coa_options.json` 생성
- `loadScenario()` 함수 구현
- 로딩 애니메이션 1.5초 적용
- 로드 완료 후 `작전계획 입력 완료` 상태 표시

완료 기준:

- 버튼 클릭 한 번으로 모든 입력 필드가 채워짐
- 발표자가 수동 입력하지 않아도 됨
- 데이터 출처가 `가상 데이터 기반 목업`으로 표시됨

## 6.2 Operation Graph Builder

목표: 계획서의 요소가 작전 그래프로 변환되는 모습을 보여준다.

노드 유형:

- 문서 노드: 작전계획서, COA 표, SOP, 기상 JSON, 통신 지도
- 임무 노드: 임무 목표, 단계, 시간 제한
- 방책 노드: A안, B안, C안
- 제약 노드: 통신 음영, 보급대기점, 차량 정비, 안개, 후송로
- 에이전트 노드: 지휘관, 참모, 현장수행, 환경, 적군
- 위험 노드: 지휘공백, 지속성 저하, 사고 대응 지연, 재판단 지연
- 결과 노드: 결심카드, 단편명령 초안

개발 항목:

- 기존 그래프 SVG 렌더링 정리
- 노드 클릭 시 오른쪽 상세 패널 표시
- `전체`, `근거`, `토론`, `실패경로`, `결심` 필터 구현
- 실패경로 모드에서는 위험 관련 노드와 edge를 강조

완료 기준:

- 그래프가 단순 장식이 아니라 근거 연결 구조로 보임
- 노드 클릭 시 “왜 이 노드가 중요한지” 설명이 뜸
- 실패경로 필터가 발표 중 바로 이해됨

## 6.3 Agent Factory

목표: 작전계획을 바탕으로 가상부대 에이전트가 생성되는 장면을 보여준다.

에이전트 계층:

- 지휘·참모 계층
  - 지휘관
  - 작전참모
  - 군수참모
  - 통신참모
  - 의무참모
  - SOP 검증
  - 레드팀
  - 심판관
- 현장 수행 계층
  - 중대장
  - 소대장
  - 분대장
  - 운전병
  - 통신병
  - 군수병
  - 정비병
- 환경·적·장비 계층
  - 적군
  - 기상
  - 지형
  - 차량
  - 통신망
  - 보급로
  - 후송로

에이전트 데이터 구조:

```json
{
  "id": "comm_agent",
  "name": "통신 에이전트",
  "layer": "staff",
  "role": "통신 음영 및 예비 중계 검토",
  "inputs": ["comm_coverage_map", "operation_plan"],
  "risk_focus": ["통신 음영", "보고 지연"],
  "status": "ready",
  "confidence": 0.82
}
```

개발 항목:

- `mock-data/agents.json` 생성
- 에이전트 생성 애니메이션 구현
- 계층별 색상 구분
- 에이전트별 역할 tooltip 구현
- 생성 완료 후 `가상부대 23개 역할 구성 완료` 표시

완료 기준:

- “참모만 만든 것이 아니라 현장 수행자까지 만들었다”는 메시지가 보임
- 에이전트가 단순 이름 목록이 아니라 역할 기반으로 보임

## 6.4 Rehearsal Engine

목표: 가상부대가 시간순으로 작전을 수행하는 듯한 리허설 화면을 구현한다.

핵심 이벤트:

```text
03:40 작전 개시
03:55 새벽 안개로 이동속도 저하
04:05 적 지연행동 발생
04:08 통신 음영구간 진입, 보고 지연
04:15 보급 대기점 부족 노출
04:22 재판단 기준 불명확으로 현장 대기
04:30 예비대 투입 판단 지연
```

개발 항목:

- `mock-data/event_timeline.json` 생성
- `runRehearsal()` 함수 구현
- 이벤트 타임라인 자동 재생
- 좌측 시간축, 중앙 지도/그래프, 우측 이벤트 상세 패널 구성
- `일시정지`, `다시보기`, `속도 1x/2x` 버튼 구현
- 이벤트 발생 시 관련 에이전트 카드 점등
- 실패경로로 이어지는 이벤트는 amber → red로 색 변화

완료 기준:

- 발표자가 버튼 한 번만 누르면 이벤트가 자동 재생됨
- 03:55, 04:08, 04:22, 04:30이 화면에 명확히 보임
- “마찰이 시간순으로 드러난다”는 메시지가 전달됨

## 6.5 Failure Path Analyzer

목표: 개별 위험요소가 아니라, 실패가 발생하는 인과 흐름을 보여준다.

대표 실패경로:

```text
최단 경로 선택
→ 적 지연행동
→ 통신 음영
→ 보고 지연
→ 재판단 기준 미명시
→ 예비대 투입 판단 지연
```

실패경로 TOP 4:

1. 지휘공백
2. 작전 지속성 저하
3. 사고 대응 지연
4. 재판단 지연

개발 항목:

- `mock-data/failure_paths.json` 생성
- 실패경로 점수화 함수 구현
- 실패경로 chain UI 구현
- 각 실패경로별 근거 문서 연결
- `단일 LLM 위험목록` 대비 보기 옵션 구현

점수 계산은 1차 MVP에서 규칙 기반으로 처리한다.

예시:

```text
위험점수 = 영향도 x 발생가능성 x 연결도 x 시간압박 보정
```

완료 기준:

- TOP 4 실패경로가 우선순위로 표시됨
- 각 실패경로가 단일 이벤트가 아니라 chain으로 표시됨
- 발표자가 “위험목록이 아니라 흐름”이라고 설명할 수 있음

## 6.6 Evidence Linker

목표: AI가 그냥 주장하는 것이 아니라 근거를 붙인다는 점을 보여준다.

근거 유형:

- 작전계획서 문장
- COA 비교표
- 통신 커버리지 지도
- 군수 현황표
- 기상 JSON
- SOP 발췌
- 에이전트 토론 로그

개발 항목:

- `mock-data/evidence_map.json` 생성
- 실패경로별 근거 badge 표시
- 결심카드 항목별 근거 링크 표시
- 근거 클릭 시 우측 패널에서 원문 preview 표시
- `근거 있음`, `추정`, `추가 확인 필요` 상태 구분

완료 기준:

- 결심카드의 각 권고 항목에 근거가 붙음
- 심판관 에이전트가 무근거 항목을 제외했다는 메시지가 보임

## 6.7 Decision Card Generator

목표: 최종 결과를 지휘관 검토용 1페이지 결심카드로 보여준다.

결심카드 구성:

- 추천 방책: B안 우선
- A안 조건부 시행 가능 여부
- 실패경로 TOP 4
- 즉시 수정안
- 재판단 기준
- 단편명령 초안
- 지휘관 확인 필요 항목

결심카드 예시:

```text
추천: B안 우선

판단 근거:
- A안은 통신 음영과 적 지연행동이 결합될 경우 지휘공백 가능성 높음
- B안은 시간은 증가하나 보급·후송 안정성이 높음

즉시 수정안:
- 통신 음영 진입 전 보고 기준 명시
- 예비 중계팀 선배치
- 보급 대기점 1개 추가
- 예비대 투입 재판단 기준 04:20 이전 지정

단편명령 초안:
- A안 시행 시 예비 통신수단 지정 후 조건부 승인
- B안을 기본 기동방책으로 준비
```

개발 항목:

- `mock-data/decision_card.json` 생성
- `mock-data/fragmentary_order.json` 생성
- 결심카드 화면 구현
- 단편명령 초안 탭 구현
- PDF/PNG export 버튼은 선택 기능으로 구현
- 화면 하단에 `AI 생성 검토안 / 지휘관 검토 필요` 표시

완료 기준:

- 최종 명령이 아니라 검토안이라는 점이 명확함
- 발표자가 마지막에 “최종 명령은 지휘관이 한다”고 말할 수 있음

## 7. 화면 구성 계획

## 7.1 화면 1: 자료 투입

목적:

- 작전계획과 제한사항이 들어오는 모습을 보여준다.

필수 UI:

- 목업 작전계획 로드 버튼
- 업로드된 파일 목록
- A/B/C 방책 표
- 제한사항 요약
- 추출된 필드 목록

핵심 문구:

> 계획서가 작전 시드로 변환됩니다.

## 7.2 화면 2: 그래프 구축

목적:

- 문서, 방책, 제약, 위험, 결심카드가 연결되는 구조를 보여준다.

필수 UI:

- 온톨로지 그래프
- 노드 상세 패널
- 필터 버튼
- 근거 연결 edge

핵심 문구:

> 문서가 아니라 관계를 봅니다.

## 7.3 화면 3: 가상부대 생성

목적:

- 가상 에이전트가 생성되는 장면을 보여준다.

필수 UI:

- 계층별 에이전트 카드
- 생성 진행률
- 역할 설명
- 준비 상태 표시

핵심 문구:

> 참모뿐 아니라 실제 수행자까지 만듭니다.

## 7.4 화면 4: 수행 리허설

목적:

- 시간순 이벤트를 자동 재생한다.

필수 UI:

- 모든 데이터는 가상 목업
- 실제 부대명, 실명, 연락처, 위치정보 없음
- 실제 군 로고, 부대 마크 없음
- 화면 어딘가에 `가상 데이터 기반 목업` 표시
- 외부 API 호출 없이도 시연 가능

## 16. 우선순위 결론

가장 먼저 만들어야 할 것은 AI 모델이 아니다.  
발표에서 먹히는 것은 아래 세 가지다.

1. **계획서가 가상부대로 바뀌는 장면**
2. **시간순 이벤트가 실패경로로 연결되는 장면**
3. **결심카드가 근거와 함께 나오는 장면**

이 세 장면이 안정적으로 보이면 데모의 설득력은 충분하다.
